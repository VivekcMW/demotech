## 1. OVERVIEW

### 1.1 Purpose & Scope

This PRD defines the **Computerized Provider Order Entry (CPOE)** system and end-to-end order lifecycle for all specialties and care settings in India. It covers unified ordering for:

- Laboratory tests
- Radiology/imaging tests
- Medications (prescribing orders)
- Procedures/interventions
- Nursing tasks
- Diet orders
- Blood/transfusion requests
- Referrals/consult requests

The goal is to provide one consistent ordering framework with safety checks, approvals, routing, tracking, and result acknowledgment.

### 1.2 Business Value

| Value | Detail |
|---|---|
| Reduced medication and test errors | Structured entry + validation + safety checks |
| Faster care turnaround | Automated routing to labs/radiology/pharmacy/nursing |
| Better interdisciplinary coordination | Single order worklist across departments |
| Better compliance and auditability | Full order trail, approvals, timestamps |
| Better patient outcomes | Critical value alerts, result acknowledgment, escalation |

### 1.3 Dependencies

| Module | Dependency |
|---|---|
| Patient Management (1.1) | Allergies, demographics, episode context |
| Scheduling (1.2) | Diagnostics/procedures/daycare appointments |
| Clinical Documentation (1.3) | Orders placed from note/encounter context |
| Platform Services (1.11) | RBAC, notifications, audit log, integration layer |
| Billing (1.6) | Charge capture from performed/dispensed orders |
| Diagnostics (Tier 1.5) | LIS/RIS/PACS execution and result publishing |
| Pharmacy (Tier 1.5) | Medication dispense workflows |

### 1.4 Out of Scope
- Detailed claim adjudication workflows
- Deep pharmacy stock management logic (covered in pharmacy PRD)
- Image viewer UX details (covered in radiology module)

## 2. USER PERSONAS & JOURNEYS

### 2.1 Personas

| Persona | Description | Needs |
|---|---|---|
| Doctor/Consultant | Creates and signs orders | Fast entry, order sets, safety alerts |
| Resident | Drafts orders under supervision | Supervision/co-sign where required |
| Nurse | Executes nursing and medication tasks | Clear worklist, administration recording |
| Lab Technician | Processes lab requests | Specimen instructions, status flow |
| Radiology Staff | Schedules and performs imaging | Prep details, protocol info |
| Pharmacist | Verifies and dispenses meds | Valid eRx, interaction alerts, substitutions |
| Dietitian | Executes diet orders | Diet type, duration, restrictions |
| Blood Bank Staff | Handles transfusion requests | Compatibility checks and urgency flags |
| Admin/Quality | Monitors order turnaround and safety | TAT dashboards, alert compliance |

### 2.2 Key Journeys

**Journey A — OPD orders from consultation**
```
Doctor opens encounter → selects order set or individual orders →
system validates allergies/interactions/duplicate tests → order sign →
routes to destination departments → patient receives instructions
```

**Journey B — IPD medication and nursing task loop**
```
Doctor enters medication + nursing orders → nurse worklist updates →
administer/perform and chart completion → exceptions/escalations tracked
```

**Journey C — Critical lab value workflow**
```
Lab validates critical result → system pushes high-priority alert to ordering doctor + nurse →
acknowledgment required → escalation to HOD if not acknowledged in SLA
```

**Journey D — Transfusion request**
```
Clinician requests blood product → blood bank compatibility check + inventory check →
issue approved unit → transfusion administration recorded → reaction monitoring
```

## 3. FUNCTIONAL REQUIREMENTS

## 3.1 Unified Order Model

**FR-ORD-001: Order Categories**
System must support these categories under one framework:
1. `LAB`
2. `RADIOLOGY`
3. `MEDICATION`
4. `PROCEDURE`
5. `NURSING_TASK`
6. `DIET`
7. `BLOOD`
8. `REFERRAL`

**FR-ORD-002: Core Order Fields**
Every order must include:
- `order_id` (UUID)
- `patient_id`, `encounter_id`, `episode_type` (OPD/IPD/ER/Daycare)
- `order_category`
- `order_item_code` + display name
- `priority` (`ROUTINE`, `URGENT`, `STAT`)
- `order_status` lifecycle
- `ordered_by`, `ordered_at`
- `signed_by`, `signed_at` (if signature required)
- `destination_department`
- `clinical_indication`
- `start_time`, `end_time` (where applicable)
- `frequency` (for repeatable orders)

**FR-ORD-003: Order States**
Common lifecycle states:
- Draft
- Signed
- Routed
- Accepted
- In Progress
- Completed
- Partially Completed
- Cancelled
- Discontinued
- Rejected

- Specialty
- Care setting (OPD/IPD/ER/ICU)

**FR-ORD-020: Allergy Checks**
For medication orders:

- Alert severity levels:
  - Informational
  - Warning
  - Hard Stop (life-threatening)
- Override only by authorized role with mandatory reason

**FR-ORD-021: Drug-Drug & Drug-Disease Interaction**

- Severity-based alerts and recommendations

**FR-ORD-022: Duplicate Test/Order Prevention**

**FR-ORD-023: Critical Risk Rules**
Configurable guardrails:
- Max dose checks by age/weight
- Renal/hepatic dose adjustment prompts
- High-alert medication second-check prompts
- Blood product compatibility constraints

## 5. WORKFLOWS

### 5.1 Standard CPOE Workflow

```text
Start order entry (encounter context) → select items/order set →
run safety validations → save draft (optional) → sign order →
route to destination queue → destination accepts and executes →
completion/result posted → clinician acknowledges (if required)
```

### 5.2 Medication Safety Workflow

```text
Enter medication order → allergy and interaction checks fire →
if severe alert: block or require privileged override →
sign and route to pharmacy/nursing → administration tracked in MAR-linked task flow
```

### 5.3 Critical Result Escalation Workflow

```text
Result marked critical by LIS/radiology verification → alert ordering clinician + nurse →
ack required within SLA → if no ack, escalate to HOD/on-call escalation chain
```

### 5.4 Order Cancellation Workflow

```text
Initiate cancel/discontinue → validate allowable state and permissions →
capture reason → notify destination queue and impacted teams →
update dependent tasks → close with audit event
```

## 1. OVERVIEW

### 1.1 Purpose & Scope

This PRD defines a single, cross-specialty CPOE (Computerized Provider Order Entry) engine for all hospitals and clinics. The engine handles the end-to-end lifecycle of clinical orders from request to execution to result acknowledgement and closure.

**Core principle:** one order engine for all order types; specialty differences are order sets/protocol configurations.

**Scope includes:**
- Order entry from all care settings (OPD/IPD/ER/ICU/daycare/teleconsult)
- Order types (labs, radiology, medications, procedures, nursing tasks, diets, blood products, referrals)
- Order sets and clinical protocols
- Decision support checks (allergy, interaction, duplication, dose/range alerts)
- Priority and turnaround handling (routine/urgent/stat)
- Fulfillment routing to departments/systems
- Result ingestion, acknowledgment, and escalation
- Cancel/hold/modify/repeat order controls
- Audit, compliance, and reporting

### 1.2 Business Value

| Value | Detail |
|---|---|
| Safer care | Reduces prescribing and ordering errors |
| Faster execution | Standardized routing and SLA-driven workflows |
| Better coordination | Unified visibility across doctors, nursing, diagnostics, pharmacy |
| Better compliance | Complete audit trail of who ordered/changed/acknowledged what |
| Better outcomes | Faster critical result action and closed-loop communication |

### 1.3 Dependencies

| Module | Dependency |
|---|---|
| Patient Management (1.1) | Patient identity, demographics, allergies, problem list |
| Scheduling (1.2) | Order-linked diagnostic/procedure slot booking |
| Clinical Documentation (1.3) | Orders initiated from note context and linked to plans |
| Platform Services (1.11) | RBAC, audit, notifications, APIs, master data |
| Billing (1.6) | Charge capture for ordered services |
| Pharmacy (1.5) | Medication fulfillment and dispensing |

- Lab analyzer technical calibration workflows
- Payer claim adjudication workflows

## 5. WORKFLOWS

### 5.1 Standard Order Workflow

```text
Create order(s) → run CDS checks → resolve alerts/overrides → sign →
release to target department queue → execution in department →
result/task completion update → result acknowledgment (if applicable) → closed
```

### 5.2 Critical Result Workflow

```text
Result marked critical → immediate alert to ordering provider + nurse →
acknowledgment required within SLA → if missed escalate to backup/HOD →
action note captured → closure logged
```

### 5.3 Medication Hold/Resume Workflow

```text
Active medication order → hold requested (reason required) →
MAR tasks paused → resume before due window OR auto-alert for prolonged hold
```

### 5.4 Order Set Workflow

```text
Select order set → evaluate conditional items → clinician confirms selections →
apply CDS checks → sign and release all relevant items together
```
