## 1. OVERVIEW

### 1.1 Purpose & Scope

This document defines the requirements for a single, specialty-agnostic scheduling engine that works across all hospital and clinic types in India. The same engine must support:

1. **OPD time-slot appointments**
2. **Daycare resource-slot scheduling** (chairs, dialysis machines, infusion bays)
3. **OT scheduling** (theatres, surgeons, anesthetists, equipment)
4. **Diagnostics scheduling** (lab sample appointments, USG/CT/MRI/echo)

The scheduling module must support front desk, call center, patient app/portal, and WhatsApp-assisted booking channels while maintaining one canonical appointment record.

### 1.2 Business Value

| Value | Detail |
|---|---|
| Reduced waiting time | Better load balancing and queue control |
| Higher utilization | Optimized doctor time, OT slots, equipment usage |
| Better patient experience | Easy reschedule/cancel, reminders, predictable flow |
| Better revenue realization | Lower no-show rates, better slot fill rate |
| Operational control | Real-time queue + delays + escalations |
| Cross-specialty consistency | One workflow for all specialties and facilities |

### 1.3 Dependencies

| Module | Dependency |
|---|---|
| Patient Management (1.1) | Patient must exist before appointment booking |
| Platform Services (1.11) | RBAC, notifications, audit log, multi-facility, APIs |
| Billing (1.6) | Appointment-linked consultation and token fees |
| Clinical Documentation (1.3) | Encounter creation at check-in/consultation start |
| Orders & CPOE (1.4) | Follow-up diagnostics/procedures scheduled from orders |
| Bed/ADT (1.7) | Admission from OPD/ER schedules and handoffs |

### 1.4 Out of Scope
- Clinical note content
- Prescription generation
- Claim adjudication logic
- OT clinical workflow details (covered in specialty overlays)

## 2. USER PERSONAS & JOURNEYS

### 2.1 Personas

| Persona | Description | Needs |
|---|---|---|
| Front Desk Executive | Books/rebooks/cancels, checks in patients | Speed, duplicate prevention, queue visibility |
| Call Center Agent | Handles inbound booking calls | Fast search and scripted booking path |
| Doctor/Consultant | Views calendar and queue | Predictable flow, buffer control, overbook approval |
| Department Coordinator | Manages specialty/daycare slots | Resource utilization and conflict resolution |
| OT Coordinator | Manages theatre blocks and teams | Multi-resource matching and pre-op readiness |
| Diagnostic Desk | Schedules scans/tests | Prep instructions and machine slot optimization |
| Patient | Self-books/reschedules/cancels | Convenience, reminders, transparency |
| Hospital Admin | Monitors SLA and utilization | KPIs, no-show, throughput, delays |

### 2.2 Key Journeys

#### Journey A: New OPD Booking (Front Desk)
```
Search patient → if not found register (PRD 1.1) → select facility/department/doctor/date →
available slots shown → choose slot → payment policy check (if token/advance) →
confirmation + reminder schedule → booking complete
```

#### Journey B: Returning Patient Self-Booking (App)
```
Login → select specialty/doctor → see available slots + fee + wait estimate →
book slot → pay if required → receive confirmation (SMS/WhatsApp/in-app)
```

#### Journey C: Daycare Slot Booking (Dialysis/Chemo)
```
Select service package → select required resources (chair/machine/nurse) →
system checks resource compatibility + duration → assign slot →
send prep instructions + checklist
```

#### Journey D: OT Scheduling
```
Procedure request approved → OT coordinator opens block calendar →
select surgeon + OT + anesthetist + equipment + bed requirement →
conflict checks + pre-op checklist validation → OT booking confirmed
```

#### Journey E: Diagnostics Scheduling from Order
```
Doctor places order → scheduling queue receives request →
diagnostics desk allocates slot based on modality + prep + urgency →
patient notified with prep instructions
```

## 3. FUNCTIONAL REQUIREMENTS

## 3.1 Scheduling Model

**FR-SCH-001: Universal Appointment Entity**

Every appointment must support:
- `appointment_id` (UUID)
- `patient_id`, `facility_id`, `department_id`
- `appointment_type` = `OPD | DAYCARE | OT | DIAGNOSTIC | TELECONSULT | FOLLOWUP`
- `channel` = `FRONT_DESK | CALL_CENTER | APP | WEB | WHATSAPP | API`
- `status` lifecycle (see section 5)
- `slot_start`, `slot_end`, `duration_minutes`
- `assigned_provider_id` (optional for non-doctor workflows)
- `assigned_resources[]` (machine/room/chair/etc.)
- `urgency` = `ROUTINE | PRIORITY | STAT`
- `visit_reason`
- `payer_context`
- `created_by`, `created_at`, `updated_by`, `updated_at`

**FR-SCH-002: Scheduling Modes**
- **Time-slot mode**: fixed/variable slot durations (5/10/15/20/30 min etc.)
- **Token/queue mode**: walk-in token sequence with expected wait calculation
- **Resource-slot mode**: capacity-based slots (e.g., 12 dialysis chairs)
- **Block scheduling mode**: reserved blocks for surgeons/departments in OT

**FR-SCH-003: Appointment Channels**
Booking must be possible via:
1. Front desk
2. Call center
3. Hospital website
4. Patient mobile app
5. WhatsApp bot/assisted channel
6. API partner channel (optional)

Channel-specific rules may differ but all produce the same canonical record.

## 3.2 Calendar, Roster, and Slot Generation

**FR-SCH-010: Provider Calendar**
- Weekly availability templates per provider
- Exceptions: leave, conferences, emergencies
- Multi-location availability (same provider across branches)
- Buffer slots configurable (before/after sessions)
- Break windows configurable

**FR-SCH-011: Slot Generation Rules**
- Slot duration configurable by specialty/provider/visit type
- Generate rolling horizon (e.g., next 90 days)
- Auto-close old slots
- Slot capacity rules for group clinics (e.g., vaccination camps)

**FR-SCH-012: Holiday and Blackout Handling**
- Facility-level holidays
- Department-level blackouts (machine maintenance)
- Emergency blackout for incident response
- Auto-notify impacted patients with rebooking options

**FR-SCH-013: Dynamic Slot Control**
- Open extra slots for high demand (with admin/doctor approval)
- Reduce slots during delays/overruns
- Auto-throttle online booking if queue exceeds threshold

## 3.3 OPD Appointment Booking

**FR-SCH-020: Booking Inputs**
- Facility
- Department/specialty
- Provider
- Visit type (`NEW`, `FOLLOWUP`, `PROCEDURE_REVIEW`)
- Preferred date/time
- Reason for visit (optional free text + coded reason)
- Payer context

**FR-SCH-021: Follow-Up Logic**
- Follow-up validity window (e.g., 7/14/30 days) configurable by provider
- Discount/zero consult fee rule support
- Follow-up appointment linked to prior encounter

**FR-SCH-022: Waitlist**
- Patient can opt for waitlist for earlier slot
- Auto-offer earliest available slot via app/WhatsApp/SMS
- Offer expiry window (e.g., 30 minutes)
- If declined/expired, offer next candidate

**FR-SCH-023: Overbooking Rules**
- Overbook caps by provider/session
- Overbook reason mandatory
- Optional approval workflow (doctor/HOD)
- Overbooked patients flagged in queue view

## 3.4 Daycare Scheduling

**FR-SCH-030: Resource-Constrained Scheduling**
For daycare procedures (dialysis/chemo/endoscopy/day surgeries):
- Require assignment of one or more resources:
  - Chair/bed/bay
  - Machine (if applicable)
  - Nurse/technician
  - Room/unit

**FR-SCH-031: Procedure Templates**
Each daycare service has:
- Standard duration
- Prep checklist
- Required resources
- Turnaround/cleaning buffer
- Recovery duration (if applicable)

**FR-SCH-032: Batch Scheduling**
- Recurring schedules (e.g., dialysis Mon-Wed-Fri)
- Cycle scheduling (e.g., chemo cycle day 1/day 8/day 15)
- Auto-book with conflict detection
- Bulk reschedule when holidays/outages occur

**FR-SCH-033: Capacity Dashboard**
- Current and future utilization by unit
- Capacity heatmap by day/hour
- Bottleneck alerts (e.g., >90% booked for 7 straight days)

## 3.5 OT Scheduling

**FR-SCH-040: OT Block Management**
- Assign OT blocks by surgeon/department/day
- Block release policy (e.g., unused block released 48 hours prior)
- Emergency OT lane separate from elective slots

**FR-SCH-041: Multi-Resource Constraint Solver**
OT booking requires availability check for:
- OT room
- Surgeon
- Assistant surgeon (optional)
- Anesthetist
- Scrub/circulating nurse
- Required equipment/instrument set
- Post-op bed/ICU if required

**FR-SCH-042: Pre-op Readiness Gates**
Booking confirmation can be gated by:
- Pre-anesthetic clearance
- Required labs done
- Blood reservation (if needed)
- Insurance/pre-auth status
- Consent readiness

**FR-SCH-043: OT Sequence Optimization**
- Sequence cases to reduce turnover loss
- Account for expected duration + cleanup buffer
- Alert when expected end spills into next slot

## 3.8 Reschedule, Cancel, No-Show

**FR-SCH-070: Reschedule**
- Allowed by channel policy (self-service cutoff e.g., 2 hours prior)
- Preserve original booking reference + reschedule history
- Re-evaluate fee/eligibility for follow-up windows

**FR-SCH-071: Cancelation**
- Reason mandatory (patient request, doctor unavailable, emergency closure)
- Cancelation fee/refund rules configurable
- Auto-notify patient and optionally auto-offer alternatives

**FR-SCH-072: No-Show Handling**
- Auto-mark no-show after configurable threshold
- No-show analytics by patient/provider/department
- Optional no-show penalty/deposit policy

## 3.10 Access, Controls, and Audit

**FR-SCH-090: RBAC**
- Front desk: create/update/check-in/cancel (within limits)
- Doctors: view own schedule, block slots, approve overbook
- Department coordinator: manage department slots/resources

**FR-SCH-091: Sensitive Flags**

## 5. WORKFLOWS (STATE MACHINES)

### 5.1 Appointment Lifecycle

```text
DRAFT → BOOKED → CONFIRMED → CHECKED_IN → WAITING → CALLED → IN_SERVICE → COMPLETED
                                 ↘
                                  RESCHEDULED (new linked BOOKED record)
                                   ↘
                                    CANCELED
BOOKED/CONFIRMED → NO_SHOW
```

### 5.2 OPD Flow

```text
Book slot → Reminder sent → Patient check-in → Queue assignment → Called → Consultation start →
Consultation end → Encounter created → Appointment completed
```

### 5.3 Daycare Flow

```text
Book service + resources → Pre-checklist validation → Arrival check-in → Resource allocation lock →
Procedure start → Recovery (if needed) → Discharge/close
```

### 5.4 OT Flow

```text
Procedure request → OT slot proposal → Multi-resource check → Pre-op gate validation →
Confirm OT booking → Day-of-surgery check-in → OT start/end timestamps → transfer/disposition
```

## 1. OVERVIEW

### 1.1 Purpose & Scope

This PRD defines the unified scheduling system used across all care settings in Indian hospitals and clinics. The same engine supports OPD consultations, diagnostics, daycare services, OT scheduling, follow-ups, teleconsults, and internal referrals.

**Core principle:** one scheduling engine, multiple scheduling modes — no separate scheduler per specialty.

**Scope includes:**
- Provider calendars and roster-based availability
- Four scheduling modes (OPD, diagnostics, daycare resource slots, OT blocks)
- Walk-in and pre-booked flows
- Queue/token management and patient check-in
- Appointment lifecycle (book → confirm → check-in → consult/procedure → checkout)
- Waitlist, reschedule, cancel, no-show handling
- Reminder/notification orchestration (SMS/WhatsApp/email/push)
- Referral and follow-up booking
- Teleconsult scheduling
- Capacity and utilization analytics

### 1.2 Business Value

| Value | Detail |
|---|---|
| Increased throughput | Reduced idle slots and better provider utilization |
| Better patient experience | Faster booking, predictable wait times, fewer missed visits |
| Better revenue capture | Fewer no-shows and improved high-value slot fill rates |
| Operational control | Real-time queue and delay tracking by department |
| Multi-setting support | Same engine for hospital OPD, clinic, dialysis, OT, diagnostics |

### 1.3 Dependencies

| Module | Dependency |
|---|---|
| Patient Management (1.1) | Patient record/UHID required before appointment completion |
| Platform Services (1.11) | RBAC, notifications, audit, multi-facility |
| Clinical Documentation (1.3) | Encounter opens from checked-in appointment |
| Orders & CPOE (1.4) | Diagnostic/procedure orders can generate bookings |
| Billing (1.6) | Appointment-linked charges, package and payer logic |
| Bed/ADT (1.7) | OT/daycare scheduling may convert to admission |

### 1.4 Out of Scope
- Clinical note capture during consultation
- Detailed billing tariff/pricing logic
- Full OT perioperative clinical workflows (covered in specialty overlays)

## 3.1 Scheduling Modes (Unified)

**FR-SCH-001: The platform shall support these 4 native scheduling modes:**

| Mode | Unit of Scheduling | Typical Use |
|---|---|---|
| OPD Time-slot | Provider time slot | General consults, follow-ups, teleconsults |

## 5. WORKFLOWS

### 5.1 Booking Workflow (Standard)

```
Select patient → choose mode/provider/resource → fetch available slots →
hold slot (short lock) → capture booking details → validate rules →
confirm booking → persist appointment → release lock → trigger notifications
```

### 5.2 Check-in Workflow

```
Find appointment/token → verify patient identity → mark arrived →
assign queue token/resource → update live board → optional pre-visit task trigger
```

### 5.3 Reschedule Workflow

```
Open existing appointment → choose new slot → validate conflicts/rules →
create linked new appointment → mark old as rescheduled → notify all parties
```

### 5.4 Waitlist Auto-offer Workflow

```
Slot opens (cancel/no-show/release) → match waitlist candidates by rules →
send offer to top candidate with expiry timer →
if accepted: book slot; else expire and offer next
```

### 5.5 OT Constraint Workflow

```
Receive OT request → check theatre + surgeon + anesthetist + equipment + bed constraints →
if all pass: schedule; else show conflict matrix with alternatives
```
