## 1. OVERVIEW

### 1.1 Purpose & Scope

This PRD defines the standards and compliance framework required to make the EHR clinically codified, interoperable, accreditation-ready, and audit-defensible for Indian healthcare settings.

It establishes product requirements for:

1. **Clinical coding standards** (ICD baseline + forward compatibility)
2. **Accreditation controls** (NABH-aligned workflows and evidence artifacts)
3. **Interoperability standards** (ABDM, FHIR, HL7, DICOM)
4. **Terminology normalization** (SNOMED CT and LOINC readiness)
5. **Privacy, legal, and audit controls** (DPDP-aligned governance)

### 1.2 Business Value

| Value | Detail |
|---|---|
| Clinical consistency | Standard codes reduce ambiguity and improve continuity |
| Regulatory readiness | Faster NABH audit preparation through built-in evidence trails |
| National interoperability | ABDM-compatible sharing under consent |
| Safer care | Standardized critical communication and safety controls |
| Better analytics | Coded data enables quality and outcomes reporting |
| Legal defensibility | Immutable records + access/share audit logs |

### 1.3 Dependencies

| Module | Dependency |
|---|---|
| 1.1 Patient Management | Demographics, ABHA linkages, consent context |
| 1.3 Clinical Documentation | Coded diagnoses/procedures and documentation integrity |
| 1.4 Orders & CPOE | Safety rules, critical result loops |
| 1.9 Diagnostics Integration | Standardized result semantics and interoperability IDs |
| 1.10 ABDM Compliance | Consent, data exchange policy enforcement |
| 1.11 Platform Services | RBAC, audit logs, governance workflow engine |

### 1.4 Out of Scope
- Accreditation application paperwork submission processes
- Third-party consulting process documents
- External portal SOPs beyond product integration requirements

## 2. USER PERSONAS & JOURNEYS

### 2.1 Personas

| Persona | Description | Needs |
|---|---|---|
| Clinician (Doctor/Nurse) | Produces and consumes coded records | Easy coding, minimal friction, compliance-safe defaults |
| Medical Records (MRD) | Coding and record quality control | Code validation, deficiency worklists, audit trace |
| Quality/NABH Coordinator | Accreditation evidence owner | Indicator dashboards, evidence exports |
| Compliance Officer/DPO | Privacy and lawful processing oversight | Access/share logs, policy controls, incident visibility |
| IT Integration Lead | Standards and interface owner | Versioned mappings, interoperability monitoring |
| Admin/HOD | Governance approver | Controlled rule and code updates |

### 2.2 Key Journeys

**Journey A — Coded clinical documentation**
```
Clinician documents encounter → selects diagnosis/procedure codes →
validation checks (version/scope/deprecation) → signs note →
coded data available for billing, analytics, and reporting
```

**Journey B — Critical communication compliance**
```
Critical result generated → clinician/nurse notified →
acknowledgment + action note captured → escalation if delayed →
audit evidence produced for quality review
```

**Journey C — NABH evidence export**
```
Quality user selects period + indicators →
system compiles evidence artifacts (timestamps, sign-offs, closure logs) →
exports accreditation package
```

**Journey D — Consent-gated data sharing**
```
External data request arrives → consent scope validation →
share allowed/denied based on policy →
event logged with purpose, actor, and data classes shared
```

## 3. STANDARDS REQUIREMENTS

## 3.1 ICD Standards (Latest Operational + Forward Readiness)

### FR-STD-001: ICD-10 Operational Baseline
The system must support ICD-10 coding in all relevant encounters with:
- Primary diagnosis
- Secondary/comorbidity diagnoses
- Provisional and final diagnosis states
- Cause-of-death coding linkage where applicable

### FR-STD-002: ICD Dictionary Governance
- Versioned ICD dictionary imports
- Effective-date activation
- Deprecated code handling (block or warn based on policy)
- Historical records retain legacy code validity

### FR-STD-003: Coding UX Requirements
- Fast search by code, synonym, and keyword
- Specialty favorites and recents
- Validation against encounter context (e.g., gender/age inconsistencies flagged)

### FR-STD-004: ICD-11 Readiness Layer
- Code-system abstraction to support multi-version coding
- Optional mapping table support (ICD-10 ↔ ICD-11 where available)
- Reporting filters by code system version

> Note: Go-live coding system remains configurable by organization policy.

---

## 3.2 Terminology Standards (Recommended Core)

### FR-STD-010: SNOMED CT Readiness
- Concept mapping layer for problem list and observation coding
- Store source code + mapped concept identifiers where available
- Maintain provenance of mapped terms

### FR-STD-011: LOINC Readiness
- Lab test master supports LOINC mapping fields
- Result payload stores both local code and standard code

### FR-STD-012: Drug Terminology Standardization
- Unified drug master with generic/brand mapping
- Controlled vocabulary support for routes/forms/strength units

---

## 3.3 Interoperability Standards

### FR-INT-001: FHIR R4 Support
- Structured data export/import compatible with FHIR R4 profiles used by national exchange requirements
- Preserve identifiers and provenance fields

### FR-INT-002: HL7 v2.x Compatibility
- Adapter support for legacy LIS/RIS integrations
- Message-level traceability and idempotent processing

### FR-INT-003: DICOM Reference Integrity
- Imaging studies linked via stable study/series/reference identifiers
- Maintain report-to-study traceability

### FR-INT-004: Interoperability Auditability
- Every exchange event logged with request/response metadata, timestamp, actor/system identity, and outcome

---

## 3.4 NABH-Aligned Product Controls

### FR-NABH-001: Documentation Integrity
- Time-stamped entries with author identity
- Signed records immutable (addendum/amendment only)
- Mandatory sections for admission/progress/discharge notes

### FR-NABH-002: Medication Safety Support
- Allergy and interaction checks integrated at order time
- High-alert medicine handling controls
- Order-to-administration traceability

### FR-NABH-003: Patient Safety Communication
- Critical value communication logs
- Acknowledgment + action capture
- Escalation policy for delayed response

### FR-NABH-004: Handover Standardization
- Standard handover templates (SBAR or configured equivalent)
- Handover completion audit trail

### FR-NABH-005: Quality Indicator Readiness
- Required indicator extraction support (configurable by facility)
- Deficiency tracking and closure workflows

### FR-NABH-006: Legal Record Readiness
- Signed printable records
- Version history and amendment reasons
- Access/disclosure logs for audit/legal release

---

## 3.5 Privacy, Legal, and Data Governance

### FR-CMP-001: Purpose-Limited Access
- Access and sharing decisions enforce role + purpose + consent constraints

### FR-CMP-002: Sensitive Record Controls
- Additional masking/access control for sensitive conditions and legal contexts
- Break-glass access requires reason and creates high-priority audit event

### FR-CMP-003: Retention Policies
- Configurable retention by record class (OPD/IPD/MLC/etc.)
- Legal hold support (records cannot be purged when hold active)

### FR-CMP-004: Disclosure Governance
- Controlled release workflow for legal and third-party requests
- Mandatory reason and approver trail

---

## 4. CONSENT & DATA SHARING REQUIREMENTS

### FR-CNS-001: Consent Artifact Model
Consent artifact must include:
- Subject (patient)
- Requester/recipient identity
- Data classes and scope
- Purpose and time validity
- Grant/revoke history

### FR-CNS-002: Runtime Consent Validation
- Every outbound data share request validated against active consent scope
- Invalid/expired consent results in deny + audit event

### FR-CNS-003: Revocation Enforcement
- Revocation effective immediately for future sharing
- Historical shared events remain immutable in audit log

### FR-CNS-004: Share Outcome Tracking
- Statuses: queued/sent/delivered/failed/denied
- Failure reason taxonomy maintained

---

## 5. GOVERNANCE & CHANGE MANAGEMENT

### FR-GOV-001: Maker-Checker for Standards Changes
Changes requiring approval:
- New code-system versions
- Mapping updates
- Critical safety rule updates
- Compliance report definition changes

### FR-GOV-002: Effective-Dated Policies
- Policy/rule updates can be scheduled
- Past records evaluated under policies active at time of action

### FR-GOV-003: Change Impact Preview
- Show estimated impact on active workflows and reports before publish

### FR-GOV-004: Rollback Controls
- Controlled rollback to last approved rule/mapping version
- Rollback action fully audited

## 7. WORKFLOWS

### 7.1 Code Assignment Workflow

```text
Open encounter → document diagnosis/procedure →
search and select code → validate code version and context →
save assignment with role (primary/secondary/etc.) →
include in summary/reports
```

### 7.2 Critical Communication Workflow

```text
Critical event generated → notify configured clinical recipients →
capture acknowledgment and action note →
if SLA breached, escalate to next level →
close event with timestamped trail
```

### 7.3 Consent-Gated Share Workflow

```text
Share request initiated → fetch active consent artifact →
validate purpose + scope + validity window →
if valid: share and log outcome
if invalid: deny and log reason
```

### 7.4 Accreditation Evidence Workflow

```text
Select indicator/report package → compile required artifacts →
validate completeness → export signed evidence package →
log export event and hash
```
