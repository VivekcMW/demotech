# PRD 1.10 — ABDM & Compliance
## Indian EHR System — National Digital Health and Regulatory Compliance

**Document ID:** PRD-CORE-1.10  
**Version:** 1.0  
**Last Updated:** June 10, 2026

## 1. Overview

Defines product requirements for Indian healthcare regulatory and interoperability compliance with focus on ABDM integration, consented data exchange, privacy/security obligations, and auditability.

### Scope
- ABHA creation/linking and verification
- Consent manager integration for data sharing
- Health Information Exchange (HIE/HIU) interactions
- PHR sharing workflows
- Compliance controls (DPDP, audit, retention, access governance)
- Facility/provider registry references where applicable

## 2. Personas
- Patient
- Front desk/registration
- Treating clinician
- Health information manager
- Compliance officer
- Integration admin

## 3. Functional Requirements

### 3.1 ABHA Lifecycle
- Link existing ABHA to patient record
- Create ABHA through approved workflow (where enabled)
- Verify and maintain ABHA linkage status
- Handle unlink/deactivation scenarios per policy

### 3.2 Consent Management
- Capture and track consent artifacts for data sharing
- Support consent grant/revoke lifecycle
- Enforce scope: data type, date range, requester, purpose
- Deny sharing when consent invalid/expired

### 3.3 Health Data Exchange
- Export clinical summaries/documents under valid consent
- Ingest external health records where authorized
- Maintain provenance metadata for imported records

### 3.4 PHR Sharing
- Push eligible records to patient-facing health record apps where consented
- Status tracking: queued/sent/delivered/failed

### 3.5 Compliance Controls
- Data minimization and purpose limitation
- Role-based access and break-glass governance
- Immutable audit logs for access/share events
- Retention and lawful archival controls

### 3.6 Compliance Monitoring
- Consent audit dashboard
- Data-sharing request outcomes
- Unauthorized access/event alerts

## 4. Data Model
- `ABHA_LINK_RECORD`
- `CONSENT_ARTEFACT`
- `CONSENT_EVENT`
- `HIE_SHARE_REQUEST`
- `HIE_SHARE_RESPONSE`
- `PHR_PUSH_EVENT`
- `COMPLIANCE_AUDIT_EVENT`

## 5. Workflows
1. Patient registration → ABHA link/create → status persisted
2. Data-share request → consent validation → share/deny with reason
3. Consent revoke → immediate access termination for future shares

## 6. Integrations
- ABDM gateway interfaces (as applicable)
- Consent manager endpoints
- HIE/FHIR interfaces
- Internal module data providers (`1.1`, `1.3`, `1.4`, `1.6`)

## 7. Reports
- ABHA linkage coverage
- Consent granted/revoked trends
- HIE share success/failure
- Access and data-sharing audit report

## 8. NFRs
- Consent validation response < 2 sec typical
- 100% audit capture for share and access events
- High availability for compliance-critical workflows

## 9. Acceptance Criteria
1. ABHA linking/verification works end-to-end with traceable status
2. Data exchange occurs only with valid consent scope
3. Consent revocation prevents subsequent unauthorized shares
4. All exchange and access events are auditable and exportable
5. Compliance dashboards reflect real transaction states
