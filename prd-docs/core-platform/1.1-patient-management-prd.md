## 1. OVERVIEW

### 1.1 Purpose & Scope

This document defines the complete requirements for the Patient Management module — the clinical center of the EHR system. Every other module (scheduling, orders, billing, nursing, pharmacy) revolves around the patient record. This module ensures:

- Every patient has a single, unique, longitudinal health record
- All clinical information from all departments is consolidated in one place
- Identity is verified and linked to national health infrastructure (ABHA)
- Every stakeholder (doctor, nurse, admin, patient themselves) sees the right view of the same data

**Scope includes:**
- Patient registration (new & returning)
- Patient identity management (UHID, ABHA, deduplication)
- Demographic management
- Patient 360° view (longitudinal record)
- Problem list, allergy registry, medication list
- Consent management
- Family linking
- Patient search & lookup
- Patient merge & deduplication
- Insurance & payer registration
- MLC (Medico-Legal Case) management

### 1.2 Business Value

| Value | Detail |
|-------|--------|
| **One patient, one record** | Eliminates duplicate records, lost history, repeated tests |
| **Continuity of care** | OBG patient's diabetic history visible to anesthetist before surgery |
| **Chronic disease management** | 10-year diabetes, BP, cardiac history in one view |
| **Safe prescribing** | Allergy + drug interaction alerts prevent adverse events |
| **ABDM compliance** | ABHA-linked records enable national health data sharing |
| **Revenue** | Returning patient recognized → avoids repeat registration fee, improves experience |

### 1.3 Dependencies

| Module | Dependency |
|--------|-----------|
| Platform Services (1.11) | Authentication, RBAC, audit logging, notifications, document store |
| Scheduling (1.2) | Patient must exist before appointment is booked |
| Clinical Documentation (1.3) | Patient record is the container for all notes |
| Orders & CPOE (1.4) | All orders linked to patient |
| Billing (1.6) | Patient + payer info required for billing |
| Bed/ADT (1.7) | Patient record required for admission |
| ABDM (1.10) | ABHA creation/linking, consent manager |

### 1.4 Out of Scope
- Clinical documentation (PRD 1.3)
- Appointment scheduling (PRD 1.2)
- Billing (PRD 1.6)
- Ayushman Bharat beneficiary verification (PRD 1.10)

## 2. USER PERSONAS & JOURNEYS

### 2.1 Personas

| Persona | Description | Key Needs |
|---------|-------------|-----------|
| **Front Desk / Registration Staff** | Creates and manages patient records | Fast registration, duplicate detection, returning patient lookup |
| **Doctor (All Specialties)** | Clinical users of patient records | Quick patient summary, complete history, safe prescribing |
| **Nurse** | Clinical care of admitted patients | Vitals history, allergy alerts, medication list |
| **MRD Staff (Medical Records)** | Manages health records, coding, legal requests | Chart retrieval, coding, completeness tracking |
| **Hospital Administrator** | Manages patient data policy | Reports, regulatory compliance, data quality |
| **Patient** | Their own health data | Access their records, consent management, ABHA |
| **Insurance/TPA Desk** | Manages insurance registration | Pre-auth, payer verification |

### 2.2 Key User Journeys

**Journey 1: New Patient — First Visit (OPD)**
```
Patient arrives → Front desk searches by name/phone →
Not found → Register as new patient →
Enter demographics → Generate UHID →
ABHA: Create new OR link existing → Photo capture →
Insurance/payer registration → Consent capture →
Direct to queue/appointment → Registration complete
Target time: < 3 minutes
```

**Journey 2: Returning Patient — Known to Hospital**
```
Patient arrives → Front desk searches (name / phone / UHID / ABHA) →
Patient found → Verify identity (name + DOB or phone OTP) →
Confirm / update demographics if changed →
Add visit purpose → Queue/appointment
Target time: < 1 minute
```

**Journey 3: Emergency / Walk-in (No Documents)**
```
Patient brought to ER → Triage nurse registers with minimal info →
Temp UHID assigned (TEMP-timestamp) →
Name (approximate / "Unknown Male" / "Unknown Female") →
Age (approximate), sex, chief complaint →
Full registration done later (by family / ward staff) →
Temp UHID merged with full record
```

**Journey 4: Doctor — Patient 360° View**
```
Doctor opens patient from appointment/ward list →
Sees: current episode summary + full history →
Checks: active problem list, allergies, current meds →
Reviews: recent vitals, last lab results, recent visits →
Opens consultation note → Past context auto-loaded into template →
Safe to prescribe: drug-allergy alert fires if relevant
```

**Journey 5: Patient Requests Record Copy**
```
Patient submits request (counter / portal / WhatsApp) →
MRD staff verifies identity →
Prepares summary (or full records) →
Patient consent for release captured →
Audit log created → Records handed/sent
```

**Journey 6: Duplicate Patient Detection & Merge**
```
Registration staff finds possible duplicate flag →
System shows: both records side by side →
Staff compares, confirms duplicate →
Selects master record →
Merge: all clinical data (notes, orders, results, bills) consolidated to master →
Merged record marked inactive → Cannot be accessed directly
Audit log: full merge history retained
```

| Field | Type | Notes |

| Father's/Husband's name | Text | Required for govt schemes |

**FR-PAT-002: UHID Generation**

- Format: `[FacilityCode]-[Year]-[SequenceNumber]`
  - Example: `APLN-2026-00045231`
- Multi-facility: UHID unique within facility; Enterprise Number (EMRN) unique across all facilities in chain
- UHID is permanent — never reassigned even if patient record deactivated
- QR code generated for UHID (for patient card printing)

**FR-PAT-003: Patient Card / Token**
- Auto-generated on registration
- Contains: Name, UHID, DOB, Blood Group, QR code
- Print-ready (label printer or A4)
- Digital version: shareable PDF via WhatsApp/SMS

**FR-PAT-004: Walk-in vs Appointment**
- Registration flow works for both walk-ins and pre-booked patients
- Pre-booked: patient found by appointment reference → confirm → check-in (no re-registration)
- Walk-in: register (if new) then add to queue

### 3.2 Patient Search & Lookup

**FR-PAT-010: Search Capabilities**

System MUST support search by:

| Search Type | Input | Behavior |
|-------------|-------|---------|
| **UHID** | Exact match | Return exact record |
| **ABHA ID** | Exact (14-digit or @address) | Return linked patient |
| **Mobile number** | Exact 10-digit | Return all matching patients |
| **Name** | Partial, phonetic | Fuzzy + phonetic match (Suresh/Suresh/Suresh spelled differently) |
| **Name + DOB** | Combined | High-confidence match |
| **Identity number** | Aadhaar last 4 / full | Return matching |
| **Appointment reference** | Appointment ID | Return patient for that appointment |
| **QR scan** | QR on patient card | Instant lookup |
| **Barcode** | UHID barcode on wristband | Instant lookup for IPD |

**FR-PAT-011: Phonetic & Fuzzy Search**
- Soundex/Metaphone algorithm for name matching
- Levenshtein distance for typo tolerance (edit distance ≤ 2)
- Results sorted by match confidence
- Minimum 3 characters to trigger search
- Results returned in < 500 ms

**FR-PAT-012: Duplicate Detection at Registration**
- Before UHID generation, system checks for potential duplicates
- Matching criteria (weighted scoring):
  - Name (phonetic match) — 30 points
  - DOB exact match — 30 points
  - Mobile match — 25 points
  - Address match — 10 points
  - Identity document match — 5 points
- Threshold: Score ≥ 60 → show "Possible duplicate" warning
- Staff can: override and create new OR select existing record
- All duplicate detections logged for quality review

### 3.3 ABHA Integration

**FR-PAT-020: ABHA Health ID**

ABHA (Ayushman Bharat Health Account) integration is mandatory for ABDM compliance.

| Scenario | Workflow |
|----------|---------|
| **Patient has ABHA** | Enter 14-digit ABHA number → verify via ABDM API → auto-pull basic demographics → link to UHID |
| **Patient doesn't have ABHA** | Offer to create during registration → Aadhaar OTP / mobile OTP verification → ABHA created → linked to UHID |
| **Patient declines ABHA** | Record noted, registration proceeds without ABHA |
| **ABHA mobile not available** | Defer to later — patient can link via patient portal/app |

**FR-PAT-021: ABHA Demographic Pull**
- On linking ABHA, auto-populate: Name, DOB, Sex, Address from ABDM
- Conflict resolution: if hospital record exists, show side-by-side → staff confirms
- Patient consent required before pulling data from ABDM

**FR-PAT-022: ABHA Health Record Sharing**
- Post-visit, discharge summary / prescription pushed to patient's ABHA-linked PHR app (with consent)
- Patient controls which facilities can share/access their records
- Consent log maintained in system (audit-compliant)

### 3.4 Patient Demographics Management

**FR-PAT-030: Update Demographics**
- Any authorized staff can update non-clinical demographics (address, phone, contact)
- Clinical staff (doctors, nurses) can update clinical demographics (blood group, allergy corrections)
- All changes: audit logged with old + new values
- Name change: requires supervisor approval + document proof
- DOB change: requires supervisor approval (impacts pediatric dosing)

**FR-PAT-031: Deceased Patient Handling**
- Mark patient as deceased: date of death, cause (ICD-10), place of death
- Deceased flag visible across all modules
- Prevent appointment booking for deceased patients
- Death certificate generation trigger
- HMIS death report integration

**FR-PAT-032: Demographic Completeness Score**
- System calculates % completeness of patient record
- Displayed to registration staff
- Minimum required for: OPD visit, IPD admission, insurance claim — different thresholds
- MRD can run "incomplete record" reports

### 3.5 Patient 360° View

This is the single most important screen in the EHR — the consolidated view of everything about a patient.

**FR-PAT-040: 360° View Sections**

#### A. Header Bar (Always Visible)
| Element | Content |
|---------|---------|
| Patient name | Full name, age, sex |
| UHID | With QR icon (click to show QR) |
| Photo | Thumbnail |
| ABHA | Linked / Not linked indicator |
| Blood group | Color-coded badge |
| Allergy alert | Red badge if allergies recorded |
| High-risk flags | Custom flags (e.g., "DNACPR", "Falls risk", "High risk pregnancy", "HIV+", "Hepatitis B") |
| Current location | Admitted ward/bed / "OPD" / "ER" / "Discharged" |

#### B. Active Problems List
- Conditions currently being managed
- Source: doctor-documented
- Each problem: ICD-10 code, description, onset date, managing specialty, status (active/resolved/inactive)
- Shared across ALL specialties — one list, visible to all departments

**FR-PAT-041: Problem List Management**
- Add problem: ICD-10 search by keyword or code
- Update status: Active → Resolved / Inactive
- Link problems across visits (same condition, multiple episodes)
- Problem list is shared, not siloed by department
- Cardiologist can see that patient is also being managed for diabetes by endocrinologist

#### C. Allergy & Adverse Reaction Registry
**FR-PAT-042: Allergy Registry**
- One allergy list per patient — shared across all departments, pharmacy, nursing
- Each entry: substance, type (drug/food/environmental/contrast), reaction type, severity (mild/moderate/severe/anaphylaxis), date identified, documented by
- Drug allergy: linked to drug master (for automated alerts in orders and pharmacy)
- Allergy status: Confirmed / Suspected / Ruled out
- No allergy known: explicitly documented (not just absence of entries)
- NKDA (No Known Drug Allergies): explicit button

**FR-PAT-043: Allergy Alert Behavior**
- Allergy alert fires when ordering a drug the patient is allergic to
- Alert levels: WARNING (mild reaction) or HARD STOP (anaphylaxis)
- Hard stop: cannot proceed without override + documented reason
- Override: requires escalated role (consultant or above) + reason entry
- All allergy-related overrides: high-priority audit log entry

#### D. Current Medications List
**FR-PAT-044: Medication List**
- Single consolidated medication list across all specialties
- Sources: prescriptions from this hospital + patient-reported medications
- Each entry: drug name, dose, frequency, route, prescribing doctor, start date, end date, indication
- Status: Active / Discontinued / Held / PRN
- Reconciliation at admission (compare home medications vs. current)
- Drug-drug interaction check across full list

#### E. Visit Timeline
**FR-PAT-045: Visit History**
- Chronological timeline of all encounters: OPD, IPD, ER, daycare, teleconsult
- Each entry: date, type, department, doctor, chief complaint, diagnoses, summary
- Click → opens full encounter record
- Filter by: department, doctor, date range, type

#### F. Vitals Summary
- Latest vitals from most recent encounter
- Trend sparklines: BP, pulse, weight, temperature over last 12 months
- Click → full vitals history chart

#### G. Recent Investigations
- Last 5 lab results (each: test, value, flag, date)
- Last 3 imaging reports (with thumbnail)
- "View all" link to full results

#### H. Recent Prescriptions
- Last 2 prescriptions (drug names, date)
- "View all" → prescription history

#### I. Upcoming Appointments
- Next 3 appointments (date, department, doctor)
- Overdue (missed) appointments highlighted

#### J. Documents
- Recent documents (discharge summaries, consent forms, old records)
- Quick upload button

#### K. Insurance & Payer Information
- Active insurance cards (company, policy number, validity, coverage amount)
- Ayushman Bharat eligibility status (verified/not verified)
- CGHS/ESI status if applicable

#### L. Social & Family Information
- Family members linked in system
- Social determinants: occupation, smoking/alcohol, residence (urban/rural)

### 3.8 Consent Management

**FR-PAT-070: Consent Types**

| Consent Type | When Captured | Format |
|-------------|---------------|--------|
| **General treatment consent** | At registration | Digital signature on tablet / e-sign |
| **Specific procedure consent** | Before procedure/surgery | Procedure-specific form (OBG, Surgery, etc.) |
| **Anesthesia consent** | Pre-op | PAC encounter |
| **MTP consent** | Before MTP | As per MTP Act (including spouse if married) |
| **Blood transfusion consent** | Before transfusion | Separate form |
| **Research / clinical trial consent** | If applicable | IRB-approved form |
| **Photography/video consent** | Before documentation | For wound/procedure photos used in records |
| **ABHA data sharing consent** | ABHA linking | Per ABDM guidelines |
| **Telemedicine consent** | First tele-consult | As per MoHFW telemedicine guidelines |
| **HIV consent (for testing)** | Before HIV test | Mandatory pre-test counseling recorded |

**FR-PAT-071: Consent Capture Methods**
1. **Tablet digital signature** — patient/guardian signs on touchscreen
2. **Aadhaar e-sign** — for legally significant documents (OTP-based)
3. **Printed + scanned** — paper form signed, scanned, uploaded
4. **Verbal consent recorded** — for emergency (documented with witness)

**FR-PAT-072: Consent Storage & Retrieval**
- All consents stored as documents linked to patient + encounter
- Retrievable by: patient, doctor, MRD, legal (with authorization)
- Tamper-evident: digital signature verified on retrieval
- Consent withdrawal: documented, access restricted, no deletion

**FR-PAT-073: Consent for Minors & Incapacitated Patients**
- Minor (< 18 years): parent/legal guardian consent mandatory
- Both signatures if possible for procedures ≥ 18 years of patient
- Incapacitated: next-of-kin or legal guardian
- Documented relationship to patient (mandatory)

### 3.9 Family Linking

**FR-PAT-080: Family Group**
- Multiple patients linked into a "family" group
- One family member (typically head of household) can manage appointments/bills for the group
- Each family member has their own independent UHID and health record (not merged)
- Family view: see all family members, their upcoming appointments, outstanding bills

**FR-PAT-081: Pediatric Parent Linking**
- Every pediatric patient (< 18) linked to parent/guardian
- Parent UHID recorded in child's record
- Parent can access child's records via portal/app (until child turns 18)
- At 18: child becomes independent, access transferred with consent

**FR-PAT-082: OBG → Newborn Linking**
- Delivery → newborn patient record auto-created
- Newborn linked to mother's UHID
- Newborn UHID generated (with "NBN" prefix for easy identification)
- Neonatal ICU transfer: UHID follows baby

### 3.11 MLC (Medico-Legal Case) Management

**FR-PAT-100: MLC Registration**

When a patient's case has medico-legal implications:

**MLC Trigger Scenarios:**
- Road traffic accident (RTA)
- Assault / violence / alleged crime
- Suspected poisoning / drug overdose
- Burns with suspicious circumstances
- Sexual assault / rape
- Unnatural / suspicious death
- Industrial accident
- Falls from height

**FR-PAT-101: MLC Documentation**
| Field | Detail |
|-------|--------|
| MLC number | Auto-generated with year-sequence (`MLC-2026-00142`) |
| Date/time of registration | Auto-captured |
| Type of MLC | Dropdown (RTA, assault, poisoning, burns, sexual assault, industrial, other) |
| Brought by | Self / Police / Public / Ambulance |
| Informant details | Name, relation, contact |
| History of event | Free text (exactly as narrated — verbatim) |
| Police station | Name, station number |
| FIR number | If police case |
| Police informed | Yes/No, Time informed |
| Magistrate informed | If applicable |
| Clinical summary | Injuries/findings (linked to clinical notes) |

**FR-PAT-102: MLC Workflow**
```
MLC scenario identified → Mark case as MLC →
MLC number assigned → MLC register entry created →
Police notification (if applicable) → Documentation starts →
All findings recorded in MLC format (verbatim, objective) →
Treating doctor signs MLC certificate →
Copy to police / court (on request, with authorization) →
MLC register closed on discharge/death →
7-year retention mandatory
```

**FR-PAT-103: MLC Confidentiality**
- MLC records: higher access restriction (doctor + admin only — not front desk)
- Disclosure to police/court: only on written request with authorization
- All MLC record access: audit logged with special "MLC_ACCESS" tag

### 3.12 Patient Merge & Deduplication

**FR-PAT-110: Merge Process**

When two UHID records are confirmed to be the same person:

| Step | Action |
|------|--------|
| 1 | Identify duplicate (system flag or manual) |
| 2 | Display side-by-side comparison: demographics, visit history, diagnoses |
| 3 | Designate master record (primary UHID to survive) |
| 4 | Review clinical data conflicts (e.g., different blood groups) — resolve before merge |
| 5 | Supervisor approves merge (cannot be undone by regular staff) |
| 6 | System merges: all records (notes, orders, results, bills, documents) linked to master UHID |
| 7 | Secondary UHID: marked "Merged into [UHID]" — read-only, permanently accessible for audit |
| 8 | Full merge audit log created |

**FR-PAT-111: Merge Constraints**
- Cannot merge if: one record has active IPD admission, active OT booking, or pending insurance claim
- These must be resolved/transferred before merge proceeds
- Merge of more than 2 records at once: not allowed (merge two at a time)
- Post-merge rollback: not possible (unmerge function reserved for Super Admin only with documented reason)

### 3.13 Patient Privacy & Confidentiality

**FR-PAT-120: Sensitive Diagnoses**
- Some diagnoses have additional access controls: HIV/AIDS, STI, psychiatric conditions, substance abuse
- Flag: "Sensitive record" — only treating team has access (not front desk or billing, unless needed)
- Patient can request entire record marked sensitive: requires CMO approval

**FR-PAT-121: VIP Patient Protocol**
- VIP flag: additional privacy controls
- Access restricted to: treating doctors + CMO + designated admin only
- No information disclosed to media, external parties
- All access events: immediate alert to hospital admin

**FR-PAT-122: Patient-requested Restrictions**
- Patient can request not to share records with specified family members
- Patient can request certain information not visible in online portal (e.g., psychiatric history)
- All restrictions documented and honored

## 5. WORKFLOWS (State Machines)

### 5.1 Patient Record States

```
NOT_REGISTERED → ACTIVE (on registration)
ACTIVE → INACTIVE (manual deactivation — rare)
ACTIVE → DECEASED (date of death entered)
ACTIVE → MERGED (merged into another record)
INACTIVE → ACTIVE (reactivation)
```

### 5.2 Allergy Alert State Machine

```
DRUG ORDERED →
Check patient allergy list →
  → No allergy found: proceed normally
  → Allergy found (substance match):
      → Severity: mild/moderate: SOFT ALERT (proceed with acknowledgment)
      → Severity: severe/life-threatening: HARD STOP
  → Cross-reactivity found:
      → CAUTION alert (proceed with acknowledgment)

SOFT ALERT:
Doctor sees alert → Acknowledges → Documents reason → Order placed
Audit log: ALLERGY_ALERT_OVERRIDDEN (low priority)

HARD STOP:
Doctor sees alert → Cannot proceed directly →
Override requires: Consultant role or above + mandatory reason text →
Pharmacist also alerted →
Audit log: ALLERGY_HARD_STOP_OVERRIDDEN (high priority alert to HOD)
```

### 5.3 Duplicate Detection Workflow

```
Registration form submitted →
Before UHID generation: run duplicate check →
  → No match (score < 40): create new record immediately
  → Low match (40-60): show as "Possible matches" — non-blocking
    Staff can dismiss and proceed
  → High match (≥ 60): show "Strong match found" — requires action
    Staff must: select existing OR explicitly choose "Create as new" with reason

If "Create as new" chosen with high match: →
  Flag both records for MRD review →
  MRD reviews within 24 hours → confirm duplicate or clear flag
```
