## 1. OVERVIEW

### 1.1 Purpose & Scope

This PRD defines the core clinical documentation system used by all specialties and care settings in Indian hospitals and clinics. It is the single engine that powers:

- Structured encounter notes (SOAP and specialty templates)
- Flowsheets (e.g., ICU hourly chart, partograph, dialysis chart)
- Clinical calculators/scores (e.g., APGAR, MELD, APACHE, Bishop)
- Narrative dictation and smart text macros
- Problem-oriented longitudinal documentation
- Procedure and discharge documentation

The engine must support OPD, IPD, ER, daycare, OT, teleconsult, and home-care extensions without module duplication.

### 1.2 Business Value

| Value | Detail |
|---|---|
| Consistent clinical quality | Standardized templates reduce variability and omissions |
| Faster documentation | Reusable templates, voice dictation, and smart macros |
| Better patient safety | Structured fields enable decision support and alerts |
| Inter-specialty continuity | One longitudinal chart shared across specialties |
| Better compliance | Complete records for NABH, legal, and audit requirements |
| Better analytics | Codified data supports quality and outcomes dashboards |

### 1.3 Dependencies

| Module | Dependency |
|---|---|
| Patient Management (1.1) | Patient identity, problem list, allergies, meds |
| Scheduling (1.2) | Encounter context from checked-in appointments |
| Platform Services (1.11) | RBAC, audit, templates governance, document store |
| Orders & CPOE (1.4) | In-note ordering actions and result linking |
| Nursing Workflows (1.8) | Shared flowsheet framework and charting events |
| ABDM/Compliance (1.10) | Structured summary export and consent handling |

### 1.4 Out of Scope
- Final billing computation
- Deep specialty-specific clinical pathways (captured in overlay specs)
- External CDS knowledgebase authoring (only integration hooks included)

## 2. USER PERSONAS & JOURNEYS

### 2.1 Personas

| Persona | Description | Needs |
|---|---|---|
| Consultant/Physician | Primary clinical author | Fast, complete, specialty-adapted notes |
| Resident/Registrar | Daily progress and procedure notes | Structured entry and supervised co-sign |
| Nurse | Vitals, MAR-linked charting, care plans | Time-based charting with minimal clicks |
| Allied Health (Physio/Dietician) | Discipline-specific notes | Reusable domain templates |
| Scribe | Assists physician documentation | Dictation + guided forms |
| Medical Records (MRD) | Review completeness and legal release | Deficiency tracking and finalization status |
| Quality/Compliance | Monitor adherence | Required field completion and signatures |

### 2.2 Key Journeys

**Journey A — OPD Consultation Note**
```
Open encounter → load specialty template → auto-fill demographics/context →
capture history/exam/assessment/plan → optional orders/prescription links →
sign and finalize → share summary to patient/PHR as per consent
```

**Journey B — IPD Daily Progress Note**
```
Open inpatient episode → create progress note from prior day copy-forward rules →
update overnight events, vitals, exam, plan → add problems/tasks → sign
```

**Journey C — ICU Flowsheet Entry**
```
Select ICU flowsheet → chart hourly parameters and interventions →
auto-trend graph + threshold alerts → handover summary generated
```

**Journey D — Discharge Summary**
```
Start discharge template → auto-pull diagnosis, procedures, key results, meds →
doctor edits narrative + follow-up advice → finalize + digitally sign →
patient copy and ABDM share (if consented)
```

## 3. FUNCTIONAL REQUIREMENTS

## 3.1 Encounter Documentation Model

**FR-CDE-001: Encounter Types**
Engine must support notes for:
- OPD consult
- ER consult
- IPD admission note
- IPD progress note
- Procedure note
- Operative note
- ICU note
- Teleconsult note
- Discharge summary
- Death summary

**FR-CDE-002: Note Lifecycle**
Statuses:
- Draft
- In Review
- Finalized
- Addendum
- Amended (with reason)
- Locked (post-discharge/legal freeze)

**FR-CDE-003: Versioning**
- Every save creates a version snapshot
- Full diff view between versions
- Finalized notes cannot be overwritten; only addendum/amendment allowed

**FR-CDE-004: Co-sign Workflow**
- Notes by trainees can require consultant co-sign
- Configurable by department/note type
- Co-sign SLA tracking and reminders

## 3.2 Template Engine

**FR-CDE-010: Template Types**
- Core templates (SOAP, Admission, Progress, Discharge)
- Specialty templates (configured via overlays)
- Procedure templates
- Smart snippet templates (micro templates/macros)

**FR-CDE-011: Template Components**
Supported field controls:
- Text (single/multi-line)
- Numeric with units and ranges
- Date/time
- Dropdown/single-select
- Multi-select checkbox
- Radio buttons
- Boolean toggles
- Calculated fields
- Repeating groups (e.g., obstetric history)
- Diagram annotations
- Attachment fields

**FR-CDE-012: Conditional Logic**
Templates support field logic:
- show/hide sections by values
- required-if conditions
- dynamic defaults based on age/sex/department
- branching sections (e.g., if pregnant → ANC fields)

**FR-CDE-013: Copy-forward Rules**
For progress notes, allow controlled carry-forward:
- Copy problem list and stable sections
- Require explicit confirmation of copied exam findings
- Visual indicator for copied vs newly edited content

**FR-CDE-014: Required Field Enforcement**
- Mandatory fields by note type and care setting
- Soft-stop (warn) or hard-stop (block finalization)
- Exception override with reason for hard-stop bypass (role restricted)

- Numeric (vitals/labs/interventions)

6. Dialysis chart container (specialty overlay extends this)

**FR-CDE-034: Alerting from Flowsheets**
- Threshold alerts to care team
- Escalation if unacknowledged within configured interval
- Alert acknowledgement log mandatory

## 5. WORKFLOWS

### 5.1 Encounter Note Authoring Workflow

```text
Start note (from appointment/episode) → load template + context data →
enter structured fields and narrative → save draft versions continuously →
run validation rules → submit for review (if needed) → finalize + sign → lock policy applies
```

### 5.2 Progress Note with Copy-forward Workflow

```text
Create progress note → fetch prior note sections eligible for copy-forward →
mark copied text visually → clinician reviews/edits → validation for stale copied fields → finalize
```

### 5.3 Flowsheet Charting Workflow

```text
Open active flowsheet instance → chart due time-cell values →
system evaluates thresholds → alert if breach → staff acknowledges and acts →
handover summary updates automatically
```

### 5.4 Deficiency Closure Workflow

```text
Deficiency identified (system or MRD) → assign to author/co-signer →
notification + due date → clinician amends/addendum →
review by MRD/quality → mark resolved
```

## 1. OVERVIEW

### 1.1 Purpose & Scope

This PRD defines a single, configurable clinical documentation engine used across all specialties, facilities, and care settings. The engine provides standardized structures (SOAP notes, forms, flowsheets, calculators, templates, smart text, voice dictation, and charting) while allowing specialty-specific overlays via configuration.

**Core principle:** One documentation engine; specialty workflows are template packs, not separate systems.

**Scope includes:**
- Encounter note framework (OPD/IPD/ER/Daycare/Teleconsult)
- Template builder and structured forms
- Flowsheet engine (time-series charting)
- Clinical scoring/calculator engine
- Smart text/macros and auto-fill
- Clinical coding integration (ICD-10, SNOMED-ready)
- Clinical sign-off, co-sign, addendum and version history
- E-signature and document locking
- Dictation (voice-to-text) and image/file embedding
- Specialty overlays (without engine fork)

| Faster documentation | Templates, macros, auto-fill and dictation reduce time |
| Safer care | Structured data supports alerts, handoffs, and continuity |

| Easy specialty expansion | New specialty support via config packs |

## 3.8 Review, Sign, Co-sign, Addendum

**FR-DOC-070: Electronic Signature**
- Sign action binds note content hash, signer identity, timestamp
- Signed note becomes non-editable; only addendum allowed

**FR-DOC-071: Co-sign Workflow**
- Roles can require mandatory co-sign (e.g., resident notes)
- Co-signer can comment and send back to draft

- Addendum cannot overwrite original content
- Must include reason and timestamp
- Linked visibly with original signed note

## 5. WORKFLOWS

### 5.1 Standard Note Authoring

```text
Open encounter → select template (auto-suggest by context) →
auto-fill context blocks → clinician enters/edits sections →
run validations/completeness checks → save draft → sign (or send for review)
```

### 5.2 Co-sign Workflow

```text
Resident drafts note → submit for co-sign → consultant reviews →
(a) approve + sign OR (b) return with comments → revise → resubmit
```

### 5.3 Signed Note Addendum

```text
Open signed note → create addendum → enter reason + content → sign addendum →
append to note timeline (original immutable)
```

### 5.4 Flowsheet Capture Loop

```text
Scheduled frequency due → reminder/task shown →
enter parameter values → validate ranges → save entries →
trend chart updates + threshold checks trigger alerts
```

### 5.5 Calculator Execution

```text
Required inputs available → run calculator (auto/manual trigger) →
store score + category + input snapshot → display interpretation text
```
