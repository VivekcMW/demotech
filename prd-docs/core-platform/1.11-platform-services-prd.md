## 1. OVERVIEW

### 1.1 Purpose & Scope

This document defines the platform-level services that underpin every module of the EHR system. These are cross-cutting technical capabilities — not visible as standalone features to end users, but required by every screen, every workflow, and every integration across all specialties, all hospital types, and all deployment sizes.

**Platform Services include:**
- User identity, roles & access control (RBAC)
- Multi-facility / multi-tenant architecture
- Audit logging & non-repudiation
- Notification engine (SMS, WhatsApp, email, in-app push)
- Document management (scan, upload, e-sign, versioning)
- Master data management (MDM)
- Reporting & analytics engine
- Telemedicine infrastructure
- Patient portal & mobile app platform
- System configuration & tenant management
- API gateway & integration bus
- Security framework

### 1.2 Business Value

| Value | Detail |
|-------|--------|
| **Single deployment, many customers** | Multi-tenant architecture enables SaaS — one codebase serves 1-bed clinic to 1000-bed hospital |
| **Compliance by default** | Audit logs, RBAC, encryption built in — not retrofitted |
| **Speed to market** | Specialty modules are built on proven platform — no re-solving auth, notifications, etc. |
| **Data integrity** | Master data management prevents inconsistencies across modules |
| **Scalability** | Horizontal scale-out handles peak load (OPD mornings, billing month-end) |

### 1.3 Dependencies
- All other PRDs depend on this document
- Must be finalized before module-level development begins

### 1.4 Out of Scope
- Clinical logic (covered in specialty overlays)
- Business logic for specific modules (covered in respective PRDs)
- Infrastructure provisioning (covered in DevOps/infra runbooks)

### 2.2 Key Admin Journeys

**Journey 1: New Hospital Onboarding**
```
Tenant creation → License activation → Facility setup → Department creation →
Master data configuration (drugs, tests, tariffs) → User creation & role assignment →
Template configuration → Integration setup (LIS, PACS, pharmacy) → Go-live
```

**Journey 2: User Role Change**
```
HR request → IT Admin receives request → Role modified → Previous access revoked →
New access granted → User notified → Audit log entry created
```

**Journey 3: Compliance Audit**
```
Auditor logs in (read-only role) → Selects date range & module → Downloads audit trail →
Views specific patient/user activity → Exports tamper-evident report
```

## 3. FUNCTIONAL REQUIREMENTS

### 3.1 User Management & Authentication

**FR-PLT-001: User Creation & Lifecycle**
- Create users with: name, email, phone, employee ID, designation, department, role(s)
- Support multiple roles per user (e.g., doctor who is also dept admin)
- User states: Active / Inactive / Locked / Pending activation
- Soft delete only — no permanent deletion (audit requirements)
- Bulk import via CSV template

**FR-PLT-002: Authentication**
- Username + password (minimum 8 chars, complexity enforced)
- OTP-based login (mobile number)
- SSO (SAML 2.0 / OAuth 2.0) for hospital AD/LDAP integration
- Biometric login support (mobile app — fingerprint, face ID)
- Session timeout configurable per role (default: 30 min inactivity)
- Concurrent session policy: configurable (allow/restrict/alert)
- IP whitelisting per tenant (optional)

**FR-PLT-003: Password Policy**
- Minimum length: 8 characters
- Must include: uppercase, lowercase, number, special character
- Password expiry: configurable (default 90 days)
- Password history: last 6 passwords not reusable
- Failed login lockout: 5 attempts → 30-min lockout (configurable)
- Forgot password: OTP to registered mobile + email
- First-time login: force password change

**FR-PLT-004: Multi-Factor Authentication (MFA)**
- MFA mandatory for: Super Admin, Hospital IT Admin, all users accessing patient data remotely
- MFA methods: TOTP (Google Authenticator), OTP SMS, hardware token
- MFA bypass: emergency access procedure (documented, audited)

### 3.2 Role-Based Access Control (RBAC)

**FR-PLT-010: Role Definitions**

Platform ships with these base roles (all customizable per tenant):

| Role Category | Base Roles |
|---------------|-----------|
| **Clinical** | Doctor, Nurse, Nurse Incharge, Paramedic, Physiotherapist, Dietitian |
| **Diagnostic** | Lab Technician, Lab Incharge, Radiologist, Sonologist, ECG Technician |
| **Pharmacy** | Pharmacist, Pharmacy Incharge |
| **Administrative** | Front Desk, Billing Executive, Insurance Executive, MRD Staff, Reception |
| **Management** | Department Admin, Hospital Admin, Hospital Director |
| **Support** | Ward Boy, Housekeeping Supervisor |
| **IT** | IT Admin, System Configuration |
| **Audit** | Compliance Officer, Auditor (read-only) |
| **Super Admin** | Vendor/platform team |

**FR-PLT-011: Permission Model**

Every permission is defined as: **Resource + Action**

Actions: `CREATE | READ | UPDATE | DELETE | APPROVE | PRINT | EXPORT | CONFIGURE`

Resources (examples):
```
patient.demographics       patient.clinical_notes     patient.vitals
patient.investigations     patient.prescriptions      patient.bills
orders.lab                 orders.radiology           orders.pharmacy
appointments.own           appointments.all           bed.assign
reports.clinical           reports.financial          admin.users
```

**FR-PLT-012: Role Inheritance**
- Roles can inherit from parent roles
- Example: `Nurse Incharge` inherits all `Nurse` permissions + adds `APPROVE` on care plans

**FR-PLT-013: Data Scope Control**
Access is scoped to:
- **Own patients** (patients doctor has seen)
- **Department patients** (all patients in doctor's department)
- **Facility patients** (all patients in facility)
- **All facilities** (for corporate/chain admins)

**FR-PLT-014: Time-Bound Access**
- Access can be granted for specific date range (locum doctors, auditors, trainees)
- Auto-revoke on expiry date
- Alert to admin 7 days before expiry

**FR-PLT-015: Break-Glass Access**
- Emergency override for critical care scenarios
- Requires reason entry (mandatory)
- Creates high-priority audit alert
- Reviewed by compliance officer within 24 hours

### 3.3 Multi-Facility / Multi-Tenant Architecture

**FR-PLT-020: Tenant Isolation**
- Complete data isolation between tenants
- No cross-tenant data access at any layer
- Separate encryption keys per tenant
- Tenant-specific configuration, masters, templates

**FR-PLT-021: Multi-Facility Support (within a tenant)**
- Single login → access to all authorized facilities
- Patient record shared across facilities (with consent)
- Separate tariffs, departments, users per facility
- Consolidated reporting across facilities
- Inter-facility referrals and transfers

**FR-PLT-022: Deployment Profiles**

| Profile | Configuration |
|---------|--------------|
| **Single clinic** | 1 tenant, 1 facility, minimal modules ON |
| **Multi-branch chain** | 1 tenant, N facilities, centralized masters |
| **Group of independent hospitals** | N tenants, optional patient record sharing (consent-based) |
| **Government cluster** | Special tenant type with ABDM mandatory, custom reporting |

**FR-PLT-023: Tenant Configuration**
- Hospital name, logo, address, contact
- Modules enabled/disabled per tenant
- Feature flags for gradual rollout
- Default language (English, Hindi, regional languages)
- Time zone (IST by default)
- Currency (INR)
- Financial year start month

### 3.4 Audit Logging & Non-Repudiation

**FR-PLT-030: Comprehensive Audit Trail**

Every action in the system generates an immutable audit log entry containing:

| Field | Value |
|-------|-------|
| `event_id` | UUID (system-generated) |
| `timestamp` | ISO 8601 with milliseconds, UTC |
| `user_id` | User who performed action |
| `user_name` | Name at time of action |
| `user_role` | Role(s) at time of action |
| `facility_id` | Facility where action occurred |
| `ip_address` | Client IP |
| `device_info` | Browser/OS/device |
| `action_type` | CREATE / READ / UPDATE / DELETE / LOGIN / LOGOUT / PRINT / EXPORT |
| `resource_type` | Patient / Order / Prescription / Bill / etc. |
| `resource_id` | Specific record ID |
| `old_value` | Previous value (for UPDATE/DELETE — JSON) |
| `new_value` | New value (for CREATE/UPDATE — JSON) |
| `session_id` | Session identifier |
| `correlation_id` | Links related events (e.g., order → result → acknowledge) |

**FR-PLT-031: Audit Log Immutability**
- Logs written to append-only store
- No UPDATE or DELETE operations on audit table
- Cryptographic hash chaining (each log entry includes hash of previous)
- Periodic export to immutable cold storage
- Tamper detection alert if chain broken

**FR-PLT-032: Audit Log Retention**
- Minimum 7 years (Indian medical records retention requirement)
- Patient-identifiable data: encrypted at rest
- Anonymized export available for analytics

**FR-PLT-033: Audit Search & Reporting**
- Search by: user, patient, date range, action type, resource type
- Filter by: department, facility, role
- Export: CSV, PDF (tamper-evident with digital signature)
- Real-time alerts for: break-glass access, bulk data export, failed logins, after-hours access to sensitive data

### 3.5 Notification Engine

**FR-PLT-040: Notification Channels**

| Channel | Use Cases |
|---------|-----------|
| **SMS** | Appointment reminders, OTP, discharge summary link, lab results ready, bill generated |
| **WhatsApp** | Appointment confirmation, prescription PDF, reports, payment receipt |
| **Email** | Appointment booking, detailed reports, billing statements |
| **In-app push (mobile)** | All of the above + clinical alerts for doctors/nurses |
| **In-app bell (web)** | Workflow notifications, order results, escalations |
| **IVR/call** | Appointment reminders (Hindi + regional languages) |

**FR-PLT-041: Notification Templates**

Each notification template includes:
- Channel (SMS/WhatsApp/email/push)
- Language (English + configurable regional)
- Trigger event (appointment confirmed, result ready, bill generated, etc.)
- Variables (patient name, appointment time, doctor name, amount, etc.)
- Character/size limits enforced
- DND (Do Not Disturb) compliance

**FR-PLT-042: Notification Rules Engine**

| Rule Type | Example |
|-----------|---------|
| **Time-based** | Appointment reminder: 24 hours before + 1 hour before |
| **Event-based** | Lab result uploaded → notify ordering doctor + patient |
| **Threshold-based** | Critical lab value → immediate alert to doctor |
| **Escalation** | Order not acknowledged in 30 min → escalate to HOD |
| **Silence** | No notifications between 10 PM - 7 AM (except critical) |

**FR-PLT-043: Notification Delivery Tracking**
- Delivery status: Sent / Delivered / Failed / Read (WhatsApp)
- Retry logic: 3 attempts for failed delivery
- Fallback channel: WhatsApp fails → SMS
- Notification log per patient/user

**FR-PLT-044: Opt-in/Opt-out Management**
- Patient preferences per channel
- TRAI DND compliance for promotional messages
- Critical clinical notifications cannot be opted out

- Max file size: 50 MB per document (configurable)

**FR-PLT-051: Document Categories**

| **Investigations** | Lab reports (external), scan reports, ECG strips |
| **Consent forms** | Surgical consent, MTP consent, research consent |
| **Identity documents** | Aadhaar, PAN, insurance cards |
| **Insurance** | Pre-auth letters, claim forms, TPA correspondence |

**FR-PLT-052: Document Versioning**

- Previous versions retained and accessible

- Active version clearly marked

**FR-PLT-053: Digital Signature & e-Sign**

**FR-PLT-054: Document Search & Retrieval**
- Search by: patient, document type, date, uploaded by
- Full-text search within PDFs (OCR-enabled)
- Related document linking (e.g., all documents for one admission episode)

### 3.7 Master Data Management (MDM)

**FR-PLT-060: Core Masters**

| Master | Key Attributes | Managed By |
|--------|---------------|------------|
| **Drug master** | Generic name, brand names, form, strength, route, schedule (H/H1/X), interactions | Platform + tenant additions |
| **Lab test master** | Test name, LOINC code, sample type, unit, normal range (age/sex-adjusted), TAT | Platform + LIS sync |
| **Radiology procedure master** | Procedure, modality, CPT code, contrast, prep instructions | Platform + tenant |
| **ICD-10 diagnosis master** | Code, description, category, chronic flag | Platform (standard) |
| **Procedure catalog** | Procedure name, type (surgical/diagnostic/therapeutic), specialties | Platform + tenant |
| **Tariff master** | Procedure → price mapping per payer (cash, TPA, Ayushman, CGHS, ESI, corporate) | Tenant |
| **Insurance/TPA master** | Payer name, type, contact, claim submission format | Platform + tenant |
| **Doctor master** | NMC registration, specialization, qualifications | Tenant (verified against NMC API) |
| **Facility master** | NABH/NHM registration, ROHINI ID, ABDM facility ID | Tenant |
| **Ward/bed master** | Ward, room, bed, class (general/semi/private/ICU), rate | Tenant |

**FR-PLT-061: Master Data Governance**
- Maker-checker workflow: any change requires approval
- Effective dates: changes take effect from specified date (no retroactive impact)
- Deactivation (not deletion): old records maintain reference integrity
- Audit trail on all master data changes
- Import via Excel/CSV with validation

**FR-PLT-062: Platform-level Master Catalog (Shared)**
- Drug master: centrally maintained by platform team, updated with every new drug approval
- ICD-10: WHO version, updated annually
- LOINC: updated with new releases
- Tenants can add local entries but cannot modify platform entries

### 3.9 Telemedicine Infrastructure

**FR-PLT-080: Video Consultation**
- WebRTC-based (no app download needed for patient — browser-based)
- Doctor: web + mobile app
- Patient: WhatsApp link → opens web consultation OR patient app
- Video quality: adaptive bitrate (works on 3G/4G)
- Features: video + audio + chat + screen share + file share (reports, images)
- Duration: configurable (default 15 min, auto-extension if doctor continues)

**FR-PLT-081: Telemedicine Workflow**
```
Appointment booked (tele-type) →
Doctor starts consultation →
Patient receives WhatsApp/SMS link →
Both join → Video consultation →
Doctor documents in EHR note (same clinical doc engine) →
e-Prescription generated →
Patient gets prescription PDF via WhatsApp →
Follow-up scheduled
```

**FR-PLT-082: Telemedicine Compliance**
- Telemedicine Guidelines 2020 (MoHFW) compliance
- Only registered medical practitioners
- e-Prescription with doctor's digital signature
- Patient identity verification (OTP-based for first visit)
- Session recording (optional, patient consent required, encrypted storage)
- Emergency escalation: if critical finding → auto-alert to nearest ER

**FR-PLT-083: Async Telemedicine**
- Text-based consultation (WhatsApp/chat)
- Photo-based consultation (dermatology, wound care)
- Report review consult (doctor reviews reports, responds with notes)

### 3.11 API Gateway & Integration Bus

**FR-PLT-100: API Standards**
- RESTful APIs (JSON) for all integrations
- FHIR R4 APIs for health data exchange (ABDM mandate)
- HL7 v2.5 for LIS/RIS integration (legacy systems)
- GraphQL for frontend-facing APIs (performance optimization)
- API versioning: URI versioning (`/api/v1/`, `/api/v2/`)

**FR-PLT-101: External Integration Types**

| Integration Type | Standards | Examples |
|-----------------|-----------|---------|
| **LIS (Lab)** | HL7 v2 (ORM/ORU) or FHIR | Siemens, Roche, Beckman, local labs |
| **RIS/PACS** | DICOM, HL7, FHIR ImagingStudy | Agfa, Carestream, local PACS |
| **Pharmacy systems** | Custom API, HL7 | Marg, Busy, Tally-based |
| **Billing/ERP** | Custom API | SAP, Oracle Financials |

| **Insurance/TPA** | FHIR, proprietary | Star Health, New India, ICICI Lombard |
| **Ayushman Bharat** | NHA APIs (REST) | Verification, pre-auth, claim submission |
| **ABDM** | FHIR R4, ABDM APIs | ABHA creation, HIE consent, PHR |

| **WhatsApp** | WhatsApp Business API | Kaleyra, Twilio, MSG91 |
| **SMS** | REST API | Twilio, MSG91, Exotel |

**FR-PLT-102: Integration Monitoring**
- API health dashboard: uptime per integration
- Message queue monitoring: pending, processed, failed
- Dead letter queue: failed messages with retry + alerting
- Integration log: every message in/out with timestamp

**FR-PLT-103: API Security**
- OAuth 2.0 + JWT tokens for all APIs
- API key for server-to-server (device integrations)
- Rate limiting: per client, per endpoint
- Input validation & sanitization (OWASP Top 10)
- TLS 1.3 minimum for all connections
- API gateway: WAF (Web Application Firewall)

### 3.12 Security Framework

**FR-PLT-110: Data Encryption**
- **At rest:** AES-256 for all patient data, separate keys per tenant
- **In transit:** TLS 1.3 minimum, HSTS enforced
- **Key management:** HSM (Hardware Security Module) or cloud KMS
- **Database:** Column-level encryption for PII (Aadhaar, phone, email)

**FR-PLT-111: Data Privacy & Consent**
- Patient consent recorded for: data collection, sharing with insurance, sharing with ABDM, research use
- Consent withdrawal: data flagged (not deleted), access restricted
- Right to access: patient can request full record download
- Data localization: all patient data stored in India (AWS Mumbai / Azure India / GCP India)

**FR-PLT-112: Application Security**
- OWASP Top 10 compliance (mandatory)
- SAST/DAST in CI/CD pipeline
- Dependency vulnerability scanning
- Penetration testing: annually + after major releases
- Bug bounty program (post-launch)

**FR-PLT-113: Infrastructure Security**
- VPC with private subnets for DB and app servers
- Bastion host only for infra access
- No direct DB access from internet
- WAF on load balancer
- DDoS protection (CloudFlare or AWS Shield)
- Intrusion detection system (IDS/IPS)

**FR-PLT-114: Business Continuity**
- RPO (Recovery Point Objective): 1 hour
- RTO (Recovery Time Objective): 4 hours
- Automated daily backups (encrypted, off-site)
- Multi-AZ deployment for HA
- Failover tested quarterly

## 5. WORKFLOWS

### 5.1 User Login Flow

```
User enters credentials →
Validate username/password →
  → Invalid: increment failed_attempts
  → 5 failures: lock account 30 min, notify user
  → Valid: proceed
Check MFA required (role-based) →
  → Yes: send OTP or prompt TOTP → Verify
  → No: proceed
Create session (JWT + refresh token) →
Load user's facilities and roles →
Redirect to default facility/dashboard →
Audit log: LOGIN event
```

### 5.2 Permission Check Flow (Every API Call)

```
API request arrives with JWT →
Validate JWT signature + expiry →
Extract user_id, tenant_id, facility_id from token →
Load user's roles for this facility (cached, 5-min TTL) →
Check: user has permission(resource_type, action) →
  → No: return 403 Forbidden + audit log UNAUTHORIZED_ACCESS
  → Yes: check data scope filter →
Apply scope filter to query →
Execute business logic →
Return response + audit log action
```

### 5.3 Notification Dispatch Flow

```
Trigger event occurs (e.g., appointment confirmed) →
Notification service subscribes to event →
Load template for (event_type, channel, language) →
Substitute variables (patient name, date, etc.) →
Check patient/user preferences (opted-out?) →
Check DND time →
  → DND: queue for morning delivery (non-critical)
  → Not DND: dispatch immediately
Dispatch to channel (SMS gateway / WhatsApp API / email / push) →
Log delivery attempt →
  → Success: update notification_log.status = delivered
  → Failure: retry (max 3 times, exponential backoff) →
  → 3 failures: fallback channel → log failure
```

### 5.4 Master Data Change Workflow (Maker-Checker)

```
Admin proposes change (new drug / tariff update / etc.) →
Change saved as "pending" with old+new values →
Notification sent to approver (senior admin / CMO / finance head) →
Approver reviews: Accept / Reject / Request modification →
  → Accept: change published with effective date → audit logged
  → Reject: change discarded → audit logged
  → Modification: back to maker
```
