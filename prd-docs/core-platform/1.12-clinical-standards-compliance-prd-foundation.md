## 1) Why this PRD is needed

This PRD will ensure the EHR is:
1. **Clinically codified** (diagnoses/procedures/labs/meds in standard formats)
2. **Accreditation-ready** (NABH-aligned workflows and evidence trails)
3. **Interoperable** (ABDM/FHIR/HL7/DICOM-based exchange)
4. **Audit-proof** (traceable and legally defensible documentation)

---

## 2) Core standards to include (Must-have)

## 2.1 ICD Coding Standards

### A. ICD-10 (Current operational baseline)
- Use ICD-10 as primary diagnosis coding baseline for all encounters.
- Support:
  - Primary diagnosis
  - Secondary/comorbidity diagnoses
  - Provisional vs final diagnosis
  - Cause-of-death coding linkage
- Requirements:
  - Search by code and keyword
  - Favorites by specialty
  - Versioned code dictionary updates
  - Deprecation handling (old code retained historically)

### B. ICD-11 Readiness (Forward compatibility)
- Add architecture to support ICD-11 transition without migration shock.
- Requirements:
  - Code system abstraction layer (ICD version as metadata)
  - Mapping table support (ICD-10 ↔ ICD-11 where available)
  - Reporting toggles by coding system
- Note: Keep ICD-11 as "readiness mode" unless policy mandates go-live.

### C. Procedure Coding (India-specific configurable approach)
- Keep procedure coding pluggable (hospital-specific package/procedure catalogs).
- Optional mapping support for CPT/ICD-PCS-style taxonomies where needed by enterprise chains.

---

## 2.2 NABH Compliance Requirements (Hospital Accreditation)

Build explicit NABH-oriented controls in workflows and evidence logs.

### A. Clinical Documentation Compliance
- Time-stamped entries with author identity
- No overwrite after final sign (addendum-only corrections)
- Mandatory sections for admission/progress/discharge
- Consent references linked to procedures

### B. Medication Safety Requirements
- Allergy and interaction checks
- High-alert medication controls
- Medication order → administration traceability
- Look-alike/sound-alike (LASA) alert support

### C. Patient Safety Goals
- Unique patient identification checks
- Critical result communication and acknowledgment tracking
- Handover standardization and audit trail
- Infection/Isolation flags in patient chart

### D. Quality Indicators & Audits
- Missing documentation dashboards
- TAT and critical alert closure reports
- Incident and variance logging hooks
- Internal audit evidence export (date/user/action history)

### E. Legal Record Readiness
- Printable and signed records
- Version history with amendment reasons
- Access and disclosure logs

> Implementation note: Keep NABH rules configurable by facility profile (Entry-level vs full NABH maturity).

---

## 2.3 ABDM & National Digital Health Compliance

- ABHA linking and verification
- Consent-based data exchange
- Data sharing only under valid scope/purpose
- FHIR-based exchange readiness
- Provenance metadata for imported records
- Audit logs for every share/access event

## 3) Additional standards to include (Strongly recommended)

## 3.1 Terminology & Coding
- **SNOMED CT readiness** for clinical concepts (problem list/observations)
- **LOINC readiness** for lab test normalization
- **Drug dictionary standardization** with local formularies and mappings

## 3.2 Interoperability Standards
- **FHIR R4** for modern API exchange
- **HL7 v2.x** compatibility for legacy LIS/RIS systems
- **DICOM** for imaging identifiers and report linkage

## 3.3 Security/Privacy/Legal
- DPDP-aligned purpose limitation and access controls
- Break-glass and sensitive chart access auditing
- Retention policies by record class (OPD/IPD/MLC/etc.)
- Export watermarking and legal release controls

## 3.4 Operational Compliance (Domain-specific)
- Blood transfusion traceability support
- MLC documentation controls
- Birth/death and statutory reporting hooks
- Specialty-specific legal forms (e.g., OBG workflows) as overlay packs

## 4) PRD scope boundaries (to avoid ambiguity)

## In Scope
- Clinical coding and terminology governance
- Accreditation-ready documentation and workflows
- Standards-based exchange/data model requirements
- Compliance dashboards, evidence artifacts, and audits

## Out of Scope
- Certification application process itself
- Third-party accreditation consulting deliverables
- Government portal operational SOP documents (only integration-ready hooks in product)

## 7) Acceptance criteria for the upcoming full PRD

1. ICD-coded diagnosis capture works with version tagging.
2. Historical records remain valid after code version updates.
3. Critical result acknowledgment is traceable end-to-end.
4. Finalized clinical records are immutable (amendment/addendum only).
5. NABH evidence exports can be generated without manual data stitching.
6. Consent validation gates all external data sharing flows.
7. FHIR/HL7/DICOM identifiers are persisted and retrievable.
8. Sensitive record access is role-controlled and fully audited.
